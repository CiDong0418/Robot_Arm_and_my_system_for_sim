1) End-to-End Task Generation Pipeline
--------------------------------------------------------------------

Stage A. User command ingestion
- Node: src/task_manager/scripts/main_input_node.py
- Input source: terminal text
- ROS topic publish: /user_command
- Responsibility: collect raw natural-language requests and push to pipeline

Stage B. High-level planning
- Node: src/task_manager/scripts/high_level_node.py
- Parser: src/task_manager/src/task_module/llm_parser.py
- Input topic: /user_command
- Output topic: /high_level_stream
- Responsibility:
  1. Convert one user command into N high-level tasks.
  2. Assign unique task id in stream order.
  3. Preserve object-flow constraints in description (initial/mid/final states).

High-level parser design highlights (llm_parser.py)
- Uses canonical location/object catalogs from domain_catalog.
- Enforces observation-gated task rule:
  - For "scan then decide" intents, first output one gate task instead of expanding both branches.
- Outputs urgency fields:
  - urgency_level in {super_urgent, priority, normal}
  - urgency_score in {10, 3, 0}

Stage C. Subtask decomposition (atomic action generation)
- Node: src/task_manager/scripts/subtask_node.py
- Parser: src/task_manager/src/task_module/subtask_parser.py
- Input topic: /high_level_stream
- Output topic: /subtasks_queue
- Log: src/task_manager/subtask_log.json

Subtask parser design highlights (subtask_parser.py)
- LLM decomposes each high-level task into atomic actions with:
  - step_id, action_type, target_object, location_id, hand_used, estimated_duration, dependencies
- MOVE_TO is forbidden in output; movement is modeled later as implicit travel cost.
- Scheduler helper actions are removed from LLM output:
  - HANDOVER, STORE_ON_TRAY, RETRIEVE_FROM_TRAY
  (these are inserted by simulator/scheduler when needed)
- Canonical hand constraints are enforced at planning prompt level:
  - OPEN_DRAWER/CLOSE_DRAWER -> Left_Arm
  - WATER_DISPENSER -> Right_Arm
  - scissors operations -> Right_Arm

Stage D. Runtime scan follow-up insertion
- Trigger path: subtask_node scan callback + scheduler fitness simulation
- Core behavior:
  - SCAN_TABLE_OBJECTS can trigger runtime branch insertion
  - Follow-up tasks are generated by scan follow-up LLM prompt in fitness.py
  - Dump file for debugging:
    - src/task_manager/scan_followup_tasks.json

2) Scheduling and Execution-Plan Construction
--------------------------------------------------------------------

Scheduler entry
- Node: src/dabc_optimizer/scripts/scheduler_node.py
- Service trigger: /start_scheduling
- Input buffer: aggregated subtasks from /subtasks_queue

Cross-run rescheduling behavior
- Scheduler attempts to pull unexecuted tasks from /get_unexecuted_tasks.
- Current policy in this version:
  - clear old hand assignment before re-optimization:
    - hand_used = None
    - hand = None
- Effect:
  - cross-run replan does not inherit old arm assignment; hands are re-decided in new optimization run.

Initial sequence generation (First LLM Bee)
- Module: src/dabc_optimizer/src/dabc_optimizer/first_llm_bee.py
- Input: task_lookup dictionary
- Output: one global initial_sequence (Task ID order)
- If LLM fails, fallback sequence = current buffered task order

Fitness and feasibility engine
- Module: src/dabc_optimizer/src/dabc_optimizer/fitness.py
- Core method: _simulate(sequence)
- Per-evaluation state is reset each time:
  - hand_holding, tray_contents, resource_clock, object_locations
- Feasibility checks include:
  - dependency correctness
  - hand availability and action-specific arm rules
  - tray capacity
  - synthetic helper insertion when needed:
    - STORE_ON_TRAY, RETRIEVE_FROM_TRAY, HANDOVER
- Objective components (weighted):
  - makespan time cost
  - travel distance
  - priority delay penalty
  - carry risk penalty

Output
- Best sequence from DABC is converted to executable plan via TaskScheduler.build_execution_plan(best_seq)
- Publish topic: /optimized_schedule
- Scheduler logs:
  - src/task_manager/scheduler_terminal_output.txt
  - src/task_manager/scheduler_log.json

3) M-DABC Method Practice (Current Version)
--------------------------------------------------------------------

Reference implementation file
- src/dabc_optimizer/src/dabc_optimizer/dabc.py

3.1 Population initialization (hybrid)
- Bee 1 uses LLM global sequence (if available).
- Remaining bees use topological+hand-state-aware generator.
- This is a hybrid seeding strategy, not pure random initialization.

3.2 Core phases
- Employed Bees: one neighborhood search per bee.
- Onlooker Bees: roulette selection by inverse fitness.
- Scout Bees: reset stagnant solution when trial counter exceeds limit.
- Periodic diversification every 20 iterations (replace worst ratio).

3.3 Neighborhood operators
- swap (prob 0.4)
- insert (prob 0.4)
- segment_shuffle (prob 0.2)
- Every mutation result is passed through repair before evaluation.

3.4 Feasibility-first repair (key M-DABC characteristic)
- _repair_topological_order enforces:
  - dependency validity
  - hand-state validity
  - free-hand-required action validity
- Current version enhances unspecified-hand modeling with aggregate held count:
  - _held_count tracks occupied capacity even when hand_used is None.

3.5 Infeasible retry repair (newly added)
- If mutated solution fitness is inf:
  - retry repair up to max_repair_attempts (current value: 6)
  - accept first feasible repaired variant
- Diagnostic fields:
  - repair_retry_success
  - repair_retry_fail

3.6 Diagnostic instrumentation
- neighbor_calls
- accepted
- rejected_infeasible
- rejected_not_better
- operator_stats (attempts/accepted/rejected breakdown by operator)
- best_parent_interleaving_ratio

This instrumentation is crucial for M-DABC tuning because it reveals whether bottleneck is:
- infeasible density problem (high rejected_infeasible), or
- local optimum problem (high rejected_not_better).

4) Differences vs Traditional DABC
--------------------------------------------------------------------

Traditional DABC (generic form)
- Usually assumes continuous vector optimization or simple permutation evaluation.
- Mutation focuses on numeric neighborhood changes.
- Feasibility often lightly handled or post-penalty only.
- No domain-specific simulator for dual-arm/tray/dynamic scan branching.

This project M-DABC (practical scheduling form)
1. Hybrid initialization
- LLM-guided first bee + constrained topological bees.

2. Domain-constrained repair
- Mutation outputs are repaired using dependency + dual-arm + tray rules.

3. Simulator-in-the-loop fitness
- Sequence is evaluated by full execution simulation (not static heuristic only).

4. Dynamic runtime branch support
- SCAN can inject follow-up tasks during simulation.

5. Infeasible-retry mechanism
- Additional re-repair loop before final rejection to raise feasible neighbor density.

6. Rich diagnostics for tuning
- Fine-grained counters to support iterative engineering decisions.

In short, this is an engineering-oriented M-DABC for constrained robotic task scheduling,
not a textbook vanilla DABC directly applied to unconstrained optimization.

5) Practical Reading of Optimization Behavior
--------------------------------------------------------------------

Observed old pattern (before stronger feasibility handling)
- Example: accepted=2, rejected_infeasible=3998, rejected_not_better=0
- Meaning:
  - almost all neighbors were infeasible
  - optimizer explored but could not enter feasible search manifold effectively

Observed improved pattern in recent run (24-task case)
- Example trend:
  - rejected_infeasible near 0
  - accepted increases continuously
  - best fitness decreases over iterations
- Interpretation:
  - feasible neighbor generation/repair became effective
  - optimizer now performs true improvement search instead of infeasible rejection loop

6) Current M-DABC Tuning Knobs
--------------------------------------------------------------------

Scheduler-level parameters
- population_size
- max_iterations
- limit

M-DABC internal parameters
- operator probability split (swap/insert/segment_shuffle)
- max_repair_attempts
- diversify interval and replace_ratio

Fitness weights
- w1, w2, w3, w4 for time/distance/delay/risk

Recommended tuning sequence
1. First minimize rejected_infeasible.
2. Then reduce rejected_not_better.
3. Finally scale max_iterations for deeper exploitation.

This order gives better efficiency than increasing iteration count blindly.
