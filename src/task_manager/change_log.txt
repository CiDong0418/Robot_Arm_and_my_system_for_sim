Update Time: 2026-04-18 (DABC Feasible-First + Adaptive Operators)

[7] File: src/dabc_optimizer/src/dabc_optimizer/dabc.py
- Added iteration timing logs every 10 iterations (shown in terminal output and diagnostics):
  - time_10iter
  - time_total
- Added feasible-first neighborhood prefilter:
  - Fast dependency/hand feasibility check before expensive fitness scoring.
  - If prefilter fails, try one repair-first pass; if still invalid, reject as infeasible early.
  - Added counter: prefilter_rejected.
- Added adaptive mutation operator probabilities:
  - Operators now selected by dynamic weights (swap/insert/segment_shuffle/parent_segment_exchange).
  - Weights are adjusted periodically from recent accepted ratio (operator recent window).
- Added parent-aware mutation operator:
  - parent_segment_exchange: swaps segments from different parent runs to optimize interleaving directly.
- Added two-stage objective logic:
  - phase1_feasibility: prioritize reducing infeasible candidates.
  - phase2_makespan: prioritize minimizing fitness/makespan after infeasible rate is reduced.
  - Phase switching is based on iteration range and recent infeasible ratio.
- Added elite diversity maintenance:
  - Keep multiple elite solutions by distinct parent interleaving signatures.
  - Diversification can re-seed from archive elites to reduce premature convergence.

Update Time: 2026-04-18 (Release-Option Cost Evaluation)

[9] File: src/dabc_optimizer/src/dabc_optimizer/fitness.py
- Replaced hardcoded tray-full fallback behavior with cost-based candidate evaluation.
- New behavior when hands are full and a hand must be released:
  - Build multiple candidate release options:
    - store_on_tray
    - deliver_to_target (direct PLACE to known future target location)
  - Estimate completion time for each candidate and choose the earliest-finishing option.
- Applied this candidate-selection flow to:
  - FREE_HAND_REQUIRED_ACTIONS deadlock handling
  - PICK no-available-hand handling
- This removes fixed-priority rule bias and lets the simulator pick the faster release path by time.

Update Time: 2026-04-20 (Softmax Release Option Sampling)

Update Time: 2026-04-20 (Stable Final Replay After Optimization)

[13] File: src/dabc_optimizer/scripts/scheduler_node.py
- Improved post-optimization execution-plan replay robustness.
- Final replay flow is now:
  - First run deterministic replay (`release_option_use_softmax = False`) to avoid random divergence from optimizer scoring.
  - If deterministic replay is infeasible, retry stochastic replay for multiple attempts.
- Added parameter:
  - `~plan_replay_retries` (default `8`) to control retry count.
- This reduces false-negative outcomes where DABC finds a finite best score but one replay attempt fails due to branch randomness.
